3 Commits with Issue Reference

Every commit should reference the issue

# The magic formula:
git commit -m "type: description #12"

# Examples:
git commit -m "fix: resolve null pointer #42"
git commit -m "feat: add user authentication #7"
git commit -m "docs: update README #15"

üìù Conventional Commits

Prefix Usage
feat: New feature
fix: Bug fix
docs: Documentation
chore: Maintenance, cleanup
refactor: Code restructuring without behavior change
test: Adding/modifying tests

üîó The Magic Keywords

These words in commit/MR automatically close issues:

Closes #12 Fixes #12 Resolves #12

Link only (without closing):

#12 Related to #12
üéØ Best Practice: Use Closes #12 in the final commit or in the MR description!

4 Create Merge Request

# Push branch

# GitLab will show a link to create the MR!

MR Description must include:

  • What was changed?
  • Closes #12 (for auto-close)
  • Review checklist

üö´ The Cardinal Sins

  1. Pushing directly to main
    ‚Üí Always use branch + MR!
  2. Commits without issue reference
    ‚Üí Always include #42 in commit!
  3. Committing secrets/passwords
    ‚Üí Use .gitignore!
  4. Huge MRs with 50 files
    ‚Üí Small, focused changes!
  5. MR without description
    ‚Üí Always include "Closes #X" + summary!

‚úÖ Pre-Merge Checklist

  • Branch created from main
  • Branch name includes issue number
  • Commits follow Conventional Commits
  • Commits reference issue (#42)
  • MR description includes Closes #42
  • No secrets in code
  • Tests pass (CI green)
  • Issue label set to workflow::review
